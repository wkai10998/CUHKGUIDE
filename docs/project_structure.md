# 项目目录说明与优化建议

## 1. 项目定位

本项目是一个基于 Flask 的课程项目，主题是“港中文申请一站式助手”。

当前已经落地的主要功能包括：
- 首页申请时间线
- 专业速查
- 申请步骤指南
- FAQ 常见问题
- 用户登录、注册、退出
- 评论功能
- AI 助手（本地检索 + Supabase 向量检索 + 智谱回答）

这个项目目前采用的是“教学友好型单体结构”：
- 入口集中
- 模板和静态资源分离
- 内容数据集中放在 JSON
- 工具函数放在 `utils/`

对于课堂展示和零基础协作来说，这种结构是合适的，因为容易找到入口，也容易讲清楚“数据怎么流动”。

## 3. 目录逐个说明

### 3.1 `app.py`

这是整个项目的主入口，也是最核心的文件。

它当前承担的职责包括：
- 创建 Flask 应用
- 读取环境变量
- 定义页面路由
- 定义登录、注册、退出、评论、资料修改等交互路由
- 维护 Session 状态
- 调用 RAG 助手逻辑
- 处理 404 和 500 错误页

如果把整个项目比作一家餐厅，`app.py` 就像总前台：
- 客户从哪里进来
- 该去哪个页面
- 提交的数据交给谁处理
- 最后返回什么结果

### 3.2 `content/`

这个目录保存的是页面直接使用的结构化内容数据。

当前文件说明：
- `stages.json`
  - 首页和申请指南中的阶段导航数据
- `guide_steps.json`
  - 每个阶段下的具体步骤、教程、提示、材料清单
- `programs.json`
  - 专业速查页面的数据
- `faq.json`
  - FAQ 页面数据
- `rag_kb.txt`
  - AI 助手的补充文本知识库

这个目录很适合零基础同学维护，因为它的门槛比改 Python 和 HTML 更低。只要理解 JSON 结构，就可以参与内容建设。

### 3.3 `templates/`

这个目录是 Flask 的 Jinja2 模板目录，负责页面结构和 HTML 展示。

可以按三类理解：

- 页面骨架
  - `base.html`
- 公共组件
  - `header.html`
  - `footer.html`
  - `login_modal.html`
- 业务页面
  - `index.html`
  - `programs.html`
  - `guide_list.html`
  - `guide.html`
  - `faq.html`
  - `faq_detail.html`
  - `assistant.html`
  - `errors/404.html`
  - `errors/500.html`

模板的价值在于“复用”：
- 统一页头页脚
- 统一颜色风格
- 不同页面共享一个骨架

### 3.4 `static/`

这个目录保存前端静态资源。

当前主要包含：
- `static/css/app.css`
  - 自定义样式
- `static/js/`
  - 页面交互逻辑
- `static/images/`
  - 页面图片和步骤截图

JS 文件按功能拆分得比较清楚：
- `programs.js`
  - 专业搜索筛选
- `faq.js`
  - FAQ 搜索筛选
- `guide.js`
  - 步骤完成状态切换、评论加载
- `components/login_modal.js`
  - 登录注册弹窗控制

### 3.5 `utils/`

这个目录是后端工具模块目录，负责把业务拆成可复用的小模块。

当前文件说明：
- `content_loader.py`
  - 统一读取 `content/*.json`
- `knowledge_base.py`
  - 将 FAQ、专业、步骤和 `rag_kb.txt` 组织成知识片段
- `rag_pipeline.py`
  - 负责切片、向量入库、检索和回答流程
- `supabase_client.py`
  - 负责用户、评论、RAG RPC 的数据库请求
- `zhipu_client.py`
  - 负责调用智谱的 embedding 和对话接口

这是目前项目里最接近“后端服务层”的部分。

### 3.6 `scripts/`

这个目录放辅助脚本，不直接负责页面展示。

当前已有：
- `ingest_rag.py`
  - 将知识库切片并写入 Supabase 向量表

它更像是“预处理工具”，不是用户点页面时执行的代码。

### 3.7 `docs/`

这个目录放项目辅助说明文档和 SQL 脚本。

当前包括：
- `presentation_notes.md`
  - 展示讲解要点
- `team_split.md`
  - 团队分工说明
- `supabase_comments.sql`
  - 用户和评论相关建表脚本
- `supabase_rag.sql`
  - RAG 向量相关 SQL

### 3.8 `tests/`

这个目录是测试目录，用来验证项目核心功能是否还正常。

当前测试覆盖了这些方向：
- 页面能否正常打开
- 登录和 Session 是否工作
- 评论和重定向逻辑是否正常
- RAG 是否有回退机制
- 内容结构是否符合预期

对于课程项目来说，已经是很加分的部分。

## 4. 当前结构的优点

### 4.1 优点一：入口清楚

新同学打开项目后，只要先看 `app.py`，就能很快知道：
- 页面有哪些
- 路由怎么走
- 数据从哪里来

### 4.2 优点二：内容和代码分开

`content/` 让数据维护和页面逻辑分离，这样：
- 内容同学可以改 JSON
- 前端同学可以改模板
- 后端同学可以改逻辑

互相不容易冲突。

### 4.3 优点三：便于课堂展示

这个项目结构不是工业级最复杂的写法，但非常适合教学：
- 容易理解
- 容易讲
- 页面功能和目录能一一对应

## 5. 当前结构存在的不足

### 5.1 `app.py` 职责偏多

现在 `app.py` 既写路由，也写认证逻辑，也写评论逻辑，还负责 AI 助手入口。

短期没问题，但如果项目继续扩展，会出现：
- 文件太长
- 修改一个地方容易影响别处
- 多人同时编辑冲突概率变高

### 5.2 `utils/` 目录偏“杂物间”

`utils/` 里同时放了：
- 数据读取
- 数据库访问
- RAG 逻辑
- 第三方 API 调用

这在小项目里常见，但长期会让同学不容易快速判断“我该去哪一个文件找问题”。

### 5.3 `docs/` 同时承担多种职责

现在 `docs/` 中既有：
- 展示稿
- 团队分工
- SQL 脚本

对当前规模没问题，但后面会让文档用途显得混杂。

### 5.4 抓取模块尚未真正落地

README 和汇报中已经提到了 `BeautifulSoup` 和数据抓取流程，但当前仓库里还没有实际抓取脚本。

这意味着你们在汇报时需要明确说明：
- 目前已完成的是“结构化内容展示 + AI 助手 + 用户交互”
- 数据抓取属于下一步可扩展模块

这样更真实，也更稳妥。

### 5.5 仓库里包含 `.idea/`

`.idea/` 属于 IDE 配置目录，通常不建议长期提交到仓库中。

原因：
- 它更偏个人开发环境
- 团队成员使用不同编辑器时价值不大
- 容易带来无关改动

## 6. 现在要不要立刻大改目录

结论：现在不建议大改。

原因很简单：
- 你们现在还是课程项目阶段
- 小组同学是零基础
- 当前结构已经能跑、能讲、能测
- 如果现在大改目录，学习成本会突然上升

比起“先改得像大厂项目”，你们更需要的是：
- 先把每块功能讲明白
- 先把汇报做顺
- 先把每个人负责的部分吃透

也就是说，现在最重要的是“可理解”和“可展示”，不是“过度重构”。

## 7. 建议的优化方向

这里给的是后续优化方向，不是要求现在立刻改。

### 7.1 方向一：把 `app.py` 拆成多个模块

未来可以考虑拆分为：
- `routes/pages.py`
- `routes/auth.py`
- `routes/comments.py`
- `routes/assistant.py`

这样每个模块职责会更清楚。

### 7.2 方向二：细分 `utils/`

未来可以进一步拆为：
- `services/`
  - 业务服务逻辑
- `clients/`
  - Supabase、智谱等第三方客户端
- `data/`
  - 内容读取、知识库构建

### 7.3 方向三：把数据分层

如果以后加入抓取脚本，建议 `content/` 不再只放最终 JSON，而是分成：

```text
content/
├── raw/       # 原始抓取结果
├── clean/     # 清洗后的中间数据
├── curated/   # 页面最终使用的数据
└── rag/       # AI 助手使用的数据
```

这样能清楚区分：
- 抓回来的原始数据
- 整理后的数据
- 页面展示数据
- AI 检索数据

### 7.4 方向四：细分 `docs/`

后面可以拆成：

```text
docs/
├── presentation/
├── database/
└── team/
```

分别存放：
- PPT 和汇报稿
- SQL 和数据库说明
- 分工和协作说明

## 9. 给小组的实际建议

对你们当前阶段来说，最好的策略不是“马上重构”，而是：

1. 先认清现在结构
2. 先按分工理解对应文件
3. 先把 PPT 和演示路线练熟
4. 等汇报完成后，再考虑轻量重构

如果要一句话总结：

当前目录结构可以继续用，而且对课堂项目是合适的。
它不是最完美，但已经足够清晰；现在最值得优化的不是代码，而是你们对代码库的理解和讲解方式。
